# Usage Simplification Plan

> 목적: 초보 운영자가 "무엇을 먼저 보고, 무엇을 입력하고, 무엇을 승인하고, 어디서 결과를 확인하는지"를 1사이클 안에서 끝낼 수 있게 제품 사용 흐름을 단순화한다.

## 1. 목표

- 초보 운영자가 로그인 후 1개의 환경 요청을 끝까지 처리할 수 있게 만든다.
- 화면마다 "지금 해야 할 다음 행동"을 1개로 제한한다.
- 생성, 검토, 승인, 실행 결과 확인을 별도 화면이 아니라 연속된 흐름으로 보이게 한다.
- 운영자가 자주 보는 정보만 먼저 드러내고, 고급 정보는 뒤로 미룬다.

## 2. 비목표

- 고급 운영자용 모든 기능을 한 번에 다 노출하지 않는다.
- 공급자 관리, 자원 상세 탐색, 감사 로그 탐색을 초보자용 첫 경로의 필수 단계로 만들지 않는다.
- 백엔드 도메인 구조를 이 문서 범위에서 재설계하지 않는다.
- 복잡한 멀티 환경 운영, 대량 배치, 고급 권한 위임은 이번 단순화 대상이 아니다.

## 3. 1사이클 최소 경로

초보 운영자가 1회차에 반드시 따라야 하는 최소 경로는 아래 순서다.

1. `Login`
2. `Dashboard`
3. `Environments`
4. `Create Environment`
5. `Plan Review`
6. `Approval Control`
7. `Job Detail`
8. `Audit` 또는 `Environment Detail`

이 경로의 원칙은 다음과 같다.

- 시작점은 항상 `Environments` 또는 `Dashboard`로 고정한다.
- `Jobs`, `Templates`, `Providers`, `Users`는 보조 경로로 내린다.
- `Job Detail`은 실행 결과와 실패 원인 확인에만 집중한다.
- `Plan Review`와 `Approval Control`은 승인 판단을 위한 전용 단계로 유지한다.

## 4. 화면별 단순화 항목

### 4.1 `Login`

P0
- 로그인 후 이동 경로를 `Dashboard` 하나로 고정한다.
- 초보자용 안내 문구를 "로그인 후 환경 생성부터 시작"으로 단순화한다.

P1
- 실패 사유를 기술 메시지 대신 짧은 사용자 문장으로 바꾼다.
- 아이디/비밀번호 외 보조 설명은 최소화한다.

P2
- 언어 전환, 세부 도움말, 추가 인증 설명은 접어 둔다.

### 4.2 `Dashboard`

P0
- 핵심 카드 3개만 남긴다: 대기 중인 검토, 승인 대기, 실패 건수.
- 첫 행동 버튼을 `환경 만들기` 하나로 고정한다.

P1
- 최근 작업 목록은 최근 3건만 보여 준다.
- 상태 색상과 라벨을 통일해 읽기 부담을 줄인다.

P2
- 공급자 요약, 사용자 요약, 감사 요약은 대시보드 기본 화면에서 제거하거나 축소한다.

### 4.3 `Environments`

P0
- 환경 목록은 이름, 상태, 마지막 실행 결과, 다음 행동만 보여 준다.
- 새 환경 생성 CTA를 화면 최상단에 고정한다.

P1
- 필터는 상태 필터 1개만 유지한다.
- 행마다 복잡한 메타데이터는 접고, 클릭 시 상세로 내려가게 한다.

P2
- 고급 검색, 정렬 옵션, 상세 비교 보기는 보조 기능으로 분리한다.

### 4.4 `Create Environment`

P0
- 초보자에게 필요한 입력만 남긴다: 환경 이름, 템플릿, 주요 네트워크 값, 인스턴스 수.
- 필수/선택 구분을 화면에서 바로 보이게 한다.
- 제출 버튼은 `검토로 이동` 하나로 고정한다.

P1
- 입력값 검증 에러는 필드 아래에 바로 표시한다.
- 고급 변수, raw JSON, 내부 식별자는 기본 접기 상태로 둔다.

P2
- 템플릿 디렉터리 탐색, 공급자 상세, 자원 상세 링크는 보조 정보로 이동한다.

### 4.5 `Plan Review`

P0
- plan 결과는 `통과`, `주의`, `실패` 3단계로 요약한다.
- 초보자가 반드시 봐야 하는 항목만 표시한다: 변경 수, 위험 신호, 실행 전 확인사항.

P1
- 긴 diff 대신 요약 카드와 핵심 변경만 먼저 보여 준다.
- 위험한 변경 항목은 강조하고, 안전한 변경은 접어 둔다.

P2
- 원시 로그, 전체 리소스 상세, 내부 경로는 디버그 영역으로 이동한다.

### 4.6 `Approval Control`

P0
- 승인 가능 여부, source job, plan 완료 여부, artifact 존재 여부를 한눈에 보여 준다.
- `승인` 버튼은 조건 충족 시에만 활성화한다.

P1
- 승인 사유 입력은 선택 항목으로 두되, 필요 시 기록 가능하게 한다.
- 승인 불가 사유는 짧고 구체적으로 표시한다.

P2
- 승인 이력의 원문 타임라인은 기본 접기 상태로 둔다.

### 4.7 `Jobs`

P0
- 초보자에게 필요한 작업만 보이게 한다: 진행 중, 완료, 실패.
- job type은 사람 친화적 이름으로 바꾼다: `환경 생성`, `검토`, `적용`.

P1
- 표는 ID 중심이 아니라 환경 이름과 결과 중심으로 정렬한다.
- 실패 job은 최상단에서 바로 찾을 수 있게 한다.

P2
- 내부 상태 코드, runner 세부 식별자, 기술적 컬럼은 접거나 상세로 내린다.

### 4.8 `Job Detail`

P0
- 상단에 상태, 대상 환경, source job, 마지막 오류, 다음 행동만 남긴다.
- 로그는 시간순으로 읽히는 하나의 스트림으로 보여 준다.

P1
- workdir, plan path, artifact 경로는 "문제 해결" 섹션으로 묶는다.
- 성공/실패에 따른 다음 버튼을 화면 하단에 명확히 배치한다.

P2
- WebSocket, raw payload, 내부 스텝 로그는 고급 보기로 분리한다.

### 4.9 `Environment Detail`

P0
- 초보자용 기본 탭은 `개요`, `진행 상태`, `다음 작업` 3개만 둔다.
- Plan -> Approval -> Apply 흐름을 한 줄 타임라인으로 보여 준다.

P1
- 실행 기록과 산출물은 접힌 패널로 제공한다.

P2
- 상세 감사, 기술 메타데이터, 전체 리소스 링크는 보조 영역으로 내린다.

### 4.10 `Templates`

P0
- 초보자에게 필요한 템플릿만 "권장"으로 강조하고 나머지는 목록화만 한다.

P1
- 템플릿 비교는 제거하고 요약 정보만 남긴다.

P2
- 디렉터리 구조와 검증 상태는 고급 사용자만 볼 수 있게 한다.

### 4.11 `Providers`

P0
- 첫 진입에서 공급자 추가 초안을 바로 노출하지 않는다.
- 운영에 필요한 연결 상태만 보여 준다.

P1
- 상세 연결과 자원 탐색은 필요할 때만 진입하도록 한다.

P2
- 공급자별 내부 리소스 트리는 고급 보기로 분리한다.

### 4.12 `Audit`

P0
- 실패 추적과 승인 추적에 필요한 기본 검색만 남긴다.

P1
- 복잡한 필터는 숨기고 최근 사건 중심으로 시작한다.

P2
- 원문 이벤트 탐색, 다중 조건 검색, 장기 보관 조회는 고급 사용 흐름으로 이동한다.

### 4.13 `Users`

P0
- 초보 운영자 기준 첫 경로에서는 사용자 관리의 존재만 보여 준다.

P1
- 일반 운영자에게는 읽기 전용으로 제한하고, 수정은 관리자 전용으로 분리한다.

P2
- 계정 생성, 비밀번호 초기화, 최근 프로비저닝 상세는 별도 관리 경로로 둔다.

## 5. 수용 기준

아래 조건을 모두 만족하면 초보 운영자 기준 단순화가 완료된 것으로 본다.

- 사용자는 로그인 후 3클릭 이내에 환경 생성 화면에 도달할 수 있다.
- 환경 생성 후 plan review와 approval control의 의미를 화면만 보고 구분할 수 있다.
- 승인 가능 여부가 불명확한 상태에서 `승인` 버튼이 노출되지 않는다.
- job detail에서 사용자는 마지막 실패 원인과 다음 행동을 바로 찾을 수 있다.
- 초보 운영자 기준 첫 화면들에서 raw 로그, 내부 식별자, 고급 설정은 기본적으로 접혀 있다.
- `Dashboard`, `Environments`, `Create Environment`, `Plan Review`, `Approval Control`, `Job Detail`만으로 1사이클을 완료할 수 있다.
- 실패 시나리오에서도 사용자는 어디서 다시 시작해야 하는지 확인할 수 있다.

## 6. 검증 시나리오

### E2E 1. 정상 생성

1. 사용자가 로그인한다.
2. `Dashboard`에서 `환경 만들기`를 누른다.
3. `Create Environment`에서 필수 값만 입력하고 검토로 이동한다.
4. `Plan Review`에서 변경 요약을 확인한다.
5. `Approval Control`에서 승인 가능 상태를 확인하고 승인한다.
6. `Job Detail`에서 apply 실행과 완료 상태를 확인한다.

기대 결과
- 사용자는 중간에 기술 화면으로 빠지지 않는다.
- 최종 결과를 하나의 흐름으로 이해한다.

### E2E 2. 입력 오류

1. 사용자가 필수 값을 비운 채 환경을 생성한다.
2. 시스템은 필드 단위 에러를 보여 준다.
3. 사용자는 어떤 값을 고쳐야 하는지 즉시 확인한다.

기대 결과
- 오류는 상단 경고가 아니라 해당 필드 바로 아래에 표시된다.
- 생성 실패 후에도 사용자는 같은 화면에서 재시도할 수 있다.

### E2E 3. 승인 불가 상태

1. 사용자가 `Approval Control`로 들어간다.
2. source job이 완료되지 않았거나 artifact가 없음을 확인한다.
3. `승인` 버튼이 비활성화되어 있음을 본다.

기대 결과
- 승인 불가 사유가 한 문장으로 설명된다.
- 버튼 상태와 설명이 서로 모순되지 않는다.

### E2E 4. plan 실패

1. 사용자가 환경을 생성한다.
2. plan이 실패한다.
3. 사용자는 `Job Detail`에서 실패 원인과 로그를 확인한다.
4. 사용자는 다시 `Create Environment` 또는 해당 환경 상세로 돌아간다.

기대 결과
- 실패 원인이 기술 로그에만 있지 않고 요약으로도 보인다.
- 다음 행동이 명시된다.

### E2E 5. 감사 확인

1. 사용자가 완료된 환경을 연다.
2. 사용자는 `Audit`에서 주요 이벤트만 확인한다.

기대 결과
- 초보 운영자는 감사 화면에서 필요한 이벤트만 빠르게 찾는다.
- 복잡한 필터를 건드리지 않아도 기본 추적이 가능하다.

## 7. 실행 원칙

- 첫 단순화는 화면 구조와 정보 우선순위부터 한다.
- 그 다음에 문구, 상태 라벨, 버튼 노출 조건을 정리한다.
- 마지막으로 고급 화면을 접거나 링크 뒤로 숨긴다.
- 한 번에 모든 화면을 바꾸지 말고, 최소 경로에 포함된 화면부터 먼저 정리한다.
