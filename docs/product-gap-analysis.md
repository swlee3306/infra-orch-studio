# Product Gap Analysis

> Historical snapshot: environment-first 전환 전 제품 격차 분석 기록이다. 현재 구현 상태 평가는 `docs/design-integration-plan.md`와 최신 API/UI 문서를 우선 기준으로 본다.

## 1. 제품 정의

이 서비스는 OpenStack 환경을 OpenTofu로 생성/관리하는 환경 단위 오케스트레이션 플랫폼이다. 제품 관점에서 핵심은 “환경을 정의하고, plan을 검토하고, 승인 후 apply를 실행하고, 상태를 추적하는 일”이다.

## 2. 현재 사용자 시나리오

현재 코드 기준으로 사용자는 다음 흐름만 부분적으로 사용할 수 있다.

1. 가입 또는 로그인
2. job 목록 확인
3. job 상세에서 상태와 로그 확인
4. API로 environment.create job 생성
5. runner가 plan/apply 수행

문제는 이 흐름이 운영자에게 보이는 방식으로 정리되어 있지 않다는 점이다.

## 3. 실서비스 관점 격차

### 사용자 가치 격차

- 사용자는 무엇을 입력해야 하는지, 어떤 상태를 확인해야 하는지, 언제 apply를 해도 되는지 알기 어렵다.
- job 목록은 식별자 중심이라 목적과 결과를 한눈에 파악하기 어렵다.
- plan 결과와 apply 대상이 UI에서 분리되어 보이지 않는다.

### 운영 안정성 격차

- apply는 민감한 작업인데 승인자/승인상태/실행 전 확인이 없다.
- 실패한 job의 복구 절차가 정의되어 있지 않다.
- artifact 유실, source job mismatch, workdir 누락 같은 운영 사고에 대한 가이드가 없다.

### 구조적 완성도 격차

- 환경 모델과 job 모델의 관계가 제품적으로 충분히 드러나지 않는다.
- plan/apply는 동작하지만 승인 가능한 서비스 객체로 다듬어져 있지 않다.
- 운영자가 보는 화면과 백엔드 상태 모델이 아직 연결되지 않았다.

## 4. 필요한 제품 보강

1. 환경 생성 흐름을 명시한다.
2. plan 검토와 승인 요청을 분리한다.
3. apply 실행은 승인 후에만 가능하게 한다.
4. 실패 시나리오를 사용자에게 보이게 한다.
5. 운영자가 상태와 로그뿐 아니라 artifact와 source 관계를 볼 수 있게 한다.

## 5. 문서 구성 제안

### 현 문서의 목적

- 이 문서는 제품 격차를 빠르게 판단하는 기준 문서가 된다.

### 포함할 항목

- 현재 상태 요약
- 목표 사용자와 운영자 시나리오
- MVP에서 멈춘 지점
- 실서비스에서 필요한 기능
- 예외/실패/승인 흐름
- 다음 스프린트 우선순위

## 6. 우선순위 제안

### 1순위

- plan/apply 승인 흐름
- 환경 생성과 job 생성의 의미 분리
- job 상세에서 실행 맥락 확인

### 2순위

- 상태/로그/artifact 가시성
- 실패 복구와 재시도
- 운영자용 화면 정리

### 3순위

- 감사/audit 정보
- 설정 분리와 배포 문서

## 7. 결론

현재 제품은 기능적으로는 초기가 만들어졌지만, 사용자가 안전하게 운영하는 데 필요한 “결정 지점”이 부족하다. 다음 단계는 기능 추가보다도 먼저 승인과 예외를 제품 언어로 고정하는 것이다.
