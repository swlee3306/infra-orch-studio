# Service Roadmap

> Historical snapshot: 서비스화 재정렬 초기안이다. 현재 구현이 이 문서의 여러 단계를 이미 지나왔으므로 active implementation checklist로 보지 않는다.

이 로드맵은 현재 코드 현실을 기준으로 서비스화 우선순위를 다시 정렬한 것이다. 기준은 사용자 가치, 운영 안정성, 구조적 완성도, 구현 난이도 대비 효과 순서다.

## Phase 0. 진단 고정

목표:

- 현재 코드와 문서의 차이를 명시한다.
- `plan/apply`와 승인 흐름이 왜 필요한지 합의한다.

완료 기준:

- 현재 상태 진단 문서가 존재한다.
- 제품 격차와 구현 백로그가 문서로 연결된다.

## Phase 1. 흐름 정합성

목표:

- job lifecycle를 사용자와 운영자 모두 이해할 수 있게 정리한다.
- `plan`과 `apply`의 의미를 분리한다.

완료 기준:

- `environment.create`, `tofu.plan`, `tofu.apply`의 역할이 문서에 일치한다.
- 승인 전/후 상태가 정의된다.
- 예외 흐름이 문서화된다.

## Phase 2. 승인 가능한 작업 흐름

목표:

- plan 검토 후 apply 승인 흐름을 서비스 기능으로 정의한다.
- 운영자가 source job과 artifact를 확인할 수 있게 한다.

완료 기준:

- plan 생성, plan 조회, apply 생성 조건이 명확하다.
- admin only apply가 제품적으로 이해 가능한 방식으로 설명된다.

## Phase 3. 운영 UI

목표:

- 잡 목록, 잡 상세, 상태, 로그, 승인 상태를 한 화면 흐름으로 제공한다.

완료 기준:

- 운영자가 API를 직접 호출하지 않아도 주요 흐름을 수행할 수 있다.
- 실패한 job의 원인을 화면에서 파악할 수 있다.

## Phase 4. 검증/안전성

목표:

- 실패 시나리오와 재시도 기준을 검증한다.
- release 전에 plan/apply 안전성을 확인한다.

완료 기준:

- 비정상 시나리오 테스트가 추가된다.
- 배포 전 체크리스트가 존재한다.

## Phase 5. 운영 배포

목표:

- dev/stage/prod에 가까운 설정 분리와 배포 기준을 만든다.

완료 기준:

- secret/config/state/workdir 정책이 문서화된다.
- Kubernetes 매니페스트가 운영 환경을 전제로 정리된다.

## Phase 6. E2E 검증

목표:

- 실제 OpenStack 환경에서 최소 1개의 환경 생성/plan/apply를 end-to-end로 검증한다.

완료 기준:

- 사전 조건, 실패 지점, rollback 기준이 문서화된다.
- 운영자가 따라 할 수 있는 절차가 남는다.

## 릴리스 게이트

각 단계의 릴리스 전에는 다음을 만족해야 한다.

- 문서가 먼저 갱신되어야 한다.
- API와 UI의 흐름이 서로 어긋나지 않아야 한다.
- 실패 시나리오가 최소 1개 이상 테스트되어야 한다.
- 운영자가 다음 행동을 예측할 수 있어야 한다.
